  PRODUCT FINDINGS
======================================================================

  Findings discovered during testing (not failures, but product issues
  worth investigating):

  [F1] ha.instance._get_rest_api() NOT IMPLEMENTED
       Severity: HIGH
       Tests: R6-B15, R6-C16
       Detail: ha_blueprint_wizard.py calls _get_rest_api() which
       does not exist on ha.instance model. Blueprint listing and
       creation features are broken. All blueprint-dependent tests
       (B02,B07-B09,B11,B16-B18,C02,C07-C08,C15,C17-C19) SKIP.

  [F2] HA label_id SLUG COLLISION
       Severity: MEDIUM
       Tests: R6-F03, R6-F11, R6-F15
       Detail: HA generates label_id from name slug. When multiple
       test labels with similar names (e.g. Chinese names that
       transliterate the same way) are created, the slug collides
       with SQL unique(label_id, ha_instance_id) constraint. No
       server-side deduplication guard exists.

  [F3] CONNECTION RESET AFTER HEAVY WEBSOCKET SYNC
       Severity: LOW
       Tests: R6-G11, R6-G12, R6-G13
       Detail: After G07's 120s full device sync via WebSocket
       controller, subsequent HTTP calls get [Errno 104] Connection
       reset by peer. Session re-authentication resolves it. This
       is an infrastructure/proxy timeout issue, not a code bug.

  [F4] disabled_by FIELD READ-ONLY ON ha.device
       Severity: INFO
       Test: R6-E07
       Detail: The disabled_by field on ha.device cannot be set
       via Odoo write(). This is expected behavior (HA controls
       device disable state), confirmed by testing.
